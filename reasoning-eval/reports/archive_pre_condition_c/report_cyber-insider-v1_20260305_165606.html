# TCP Three-Way Handshake & Connection Termination ## Three-Way Handshake (Connection Establishment) The handshake synchronizes sequence numbers and establishes a bidirectional connection: ``` Client Server | | |------------ SYN (seq=x) -------------> | | | | <------ SYN-ACK (seq=y, ack=x+1) ----- | | | |-------- ACK (seq=x+1, ack=y+1) -------> | | | |========== Connection Established ======= | ``` ### Step-by-Step Breakdown **1. SYN (Client → Server)** ``` - Client sends SYN packet with: - SYN flag = 1 - Sequence number = x (random initial value) - No data payload - Client enters SYN_SENT state ``` **2. SYN-ACK (Server → Client)** ``` - Server receives SYN, responds with: - SYN flag = 1 (server also initiating its sequence) - ACK flag = 1 (acknowledging client's sequence) - Sequence number = y (server's random initial value) - Acknowledgment number = x + 1 (confirming receipt) - Server enters SYN_RECEIVED state ``` **3. ACK (Client → Server)** ``` - Client receives SYN-ACK, sends final ACK: - SYN flag = 0 - ACK flag = 1 - Sequence number = x + 1 - Acknowledgment number = y + 1 - Client enters ESTABLISHED state - Server receives ACK and enters ESTABLISHED state ``` ## Connection Termination (Four-Way Handshake) TCP uses a graceful shutdown with FIN packets: ``` Client Server | | |------------ FIN (seq=x) -------------> | | | | <---------- ACK (ack=x+1) ---------- | | | | <---------- FIN (seq=y) ---------- | | | |------------ ACK (ack=y+1) ------------->| | | |========== Connection Closed =========== | ``` ### Detailed Steps **1. FIN (Client → Server)** ``` - Client initiates close: - FIN flag = 1 - Sequence number = x - Client enters FIN_WAIT_1 state - Client can no longer send data ``` **2. ACK (Server → Client)** ``` - Server acknowledges: - ACK flag = 1 - Acknowledgment number = x + 1 - Server enters CLOSE_WAIT state - Server can still send data ``` **3. FIN (Server → Client)** ``` - Server sends its own FIN: - FIN flag = 1 - Sequence number = y - Server enters LAST_ACK state ``` **4. ACK (Client → Server)** ``` - Client acknowledges server's FIN: - ACK flag = 1 - Acknowledgment number = y + 1 - Client enters TIME_WAIT state (waits 2 MSL) - Server enters CLOSED state - Client finally enters CLOSED state ``` ## Key Protocol Details ### Sequence Numbers ```python # Why sequence numbers matter: - Prevent packet reordering issues - Detect duplicate packets - Enable reliable delivery - Each direction has independent sequence space # Example: Client seq: 1000, Server seq: 5000 After handshake: - Client can send data with seq starting at 1001 - Server can send data with seq starting at 5001 ``` ### Flags in TCP Header ``` | URG | ACK | PSH | RST | SYN | FIN | | 1 | 1 | 1 | 1 | 1 | 1 | (bit positions) - SYN: Synchronize sequence numbers (handshake) - ACK: Acknowledgment number is valid - FIN: No more data from sender - RST: Reset connection (error/abort) - PSH: Push data to application immediately - URG: Urgent data present ``` ## State Diagram (Simplified) ``` Client Side: CLOSED → SYN_SENT → ESTABLISHED → FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT → CLOSED Server Side: CLOSED → LISTEN → SYN_RECEIVED → ESTABLISHED → CLOSE_WAIT → LAST_ACK → CLOSED ``` ## Why TIME_WAIT? # TCP Three-Way Handshake & Connection Termination ## Three-Way Handshake (Connection Establishment) The handshake synchronizes sequence numbers and establishes a bidirectional connection: ``` Client Server | | |------------ SYN (seq=x) ------------->| | (SYN_SENT state) | | (LISTEN state) | | |<----- SYN-ACK (seq=y, ack=x+1) --------| | (ESTABLISHED state) (SYN_RECEIVED) | | |------- ACK (seq=x+1, ack=y+1) ------->| | (ESTABLISHED state) (ESTABLISHED) | | ``` ### Step-by-Step Breakdown **1. SYN (Synchronization)** ``` Client → Server - Sets SYN flag = 1 - Chooses initial sequence number (ISN): seq=x - Typically sends MSS (Maximum Segment Size) option - No data payload ``` **2. SYN-ACK (Synchronization Acknowledgment)** ``` Server → Client - Sets SYN flag = 1, ACK flag = 1 - Chooses its own ISN: seq=y - Acknowledges client's sequence: ack=x+1 - Also sends MSS option ``` **3. ACK (Acknowledgment)** ``` Client → Server - Sets ACK flag = 1 - Echoes server's sequence: ack=y+1 - Can include data payload - Connection is now ESTABLISHED on both sides ``` ## Connection Termination (Four-Way Handshake) ``` Client Server | | |------------ FIN (seq=x) ------------->| | (FIN_WAIT_1 state) | | (CLOSE_WAIT) | | |<----- ACK (seq=y, ack=x+1) ------------| | (FIN_WAIT_2 state) | | | |<----- FIN (seq=y, ack=x+1) ------------| | (TIME_WAIT state) (LAST_ACK) | | |------------ ACK (seq=x+1) ------------>| | (CLOSED state) (CLOSED) | | ``` ### FIN Packet Details **1. # Email Authentication Protocols These three work in layers, each addressing different spoofing vulnerabilities: ## SPF (Sender Policy Framework) **What it does:** Publishes a DNS record listing which mail servers are authorized to send email from your domain. **How it works:** - You add a TXT record to DNS: `v=spf1 ip4:192.0.2.0 include:_spf.google.com ~all` - Receiving server checks: "Is this email's origin server on the approved list?" - If not, it can reject or flag the message **Limitations:** - Only validates the envelope sender (not the visible "From" address) - Doesn't prevent someone from spoofing your domain if they control a server on your SPF list - Breaks with email forwarding ## DKIM (DomainKeys Identified Mail) **What it does:** Cryptographically signs email content using public/private key pairs. **How it works:** - Sender's server signs the message with a private key - Public key is published in DNS - Receiver verifies the signature hasn't been tampered with - Proves the domain actually sent it (or authorized someone to) **Advantages:** - Survives forwarding better than SPF - Harder to forge (requires the private key) **Limitations:** - Doesn't tell you *what* to do if verification fails - Doesn't authenticate the visible "From" header ## DMARC (Domain-based Message Authentication, Reporting and Conformance) **What it does:** Sets policy for what happens when SPF/DKIM fail, and reports results. **How it works:** - You publish a policy: `v=DMARC1; p=reject; rua=mailto:reports@example.com` - Receiver checks if email passes SPF *or* DKIM - If it fails, the policy says: reject, quarantine, or allow - Receiver sends reports back to you **Key advantage:** Closes the gap by actually *enforcing* authentication ## Why All Three? | Protocol | Validates | Enforces | |----------|-----------|----------| | SPF | Server IP | No | | DKIM | Content signature | No | | DMARC | SPF+DKIM alignment | Yes | A spoofed email might pass SPF (if sent from an authorized server) but fail DKIM. DMARC ensures at least one passes *and* aligns with the visible domain. ## Real Limitations - **Adoption gaps:** Not all domains publish these records - **Configuration errors:** Weak policies (like `p=none`) don't actually block anything - **Display name spoofing:** These don't prevent "From: Your Bank <attacker@evil.com>" - **Subdomain attacks:** Attacker can use `mail.yourbank.com` if not explicitly restricted - **User behavior:** Even authenticated emails can be phishing These are necessary but not sufficient—they work best alongside user training and other defenses.

# Email Authentication Protocols These three work in layers, each addressing different spoofing vulnerabilities: ## SPF (Sender Policy Framework) **What it does:** Specifies which mail servers are authorized to send email for your domain. **How:** You publish a DNS record listing IP addresses allowed to send mail as `@yourdomain.com`. When an email arrives, the recipient's server checks: "Does the sending server's IP match the SPF record?" **Limitation:** Only validates the envelope sender (technical "from"), not the display name users see. Easy to spoof if SPF is missing or misconfigured. ## DKIM (DomainKeys Identified Mail) **What it does:** Cryptographically signs email content so recipients can verify it hasn't been altered and actually came from your domain. **How:** Your mail server signs outgoing messages with a private key. Recipients use your public key (in DNS) to verify the signature. Changes to the email break the signature. **Limitation:** Doesn't prevent someone from sending *new* emails claiming to be you—only proves this specific message is legitimate if they send it through your server. ## DMARC (Domain-based Message Authentication, Reporting and Conformance) **What it does:** Ties SPF and DKIM together and tells recipients what to do if authentication fails. **How:** You publish a policy saying "require SPF *and* DKIM" or "require either one." You also specify: reject suspicious mail, quarantine it, or just monitor (report-only mode). **Real power:** Gives you visibility into who's sending mail as your domain and lets you block imposters. ## The Gaps Even together, these don't catch: - **Display name spoofing** ("From: Your Bank" with a spoofed address) - **Lookalike domains** (paypa1.com vs paypal.com) - **Compromised legitimate accounts** (real authentication, malicious intent) - **Phishing from authorized servers** (if your employee's account is hacked) They're necessary but not sufficient—user awareness still matters.
