# TCP Three-Way Handshake & Connection Termination ## Three-Way Handshake (Connection Establishment) The handshake synchronizes sequence numbers and establishes a bidirectional connection: ``` Client Server | | |------------ SYN (seq=x) ------------->| | (SYN_SENT state) | | (LISTEN state) | | |<----- SYN-ACK (seq=y, ack=x+1) --------| | (ESTABLISHED state) (SYN_RECEIVED) | | |------- ACK (seq=x+1, ack=y+1) ------->| | (ESTABLISHED state) (ESTABLISHED) | | ``` ### Step-by-Step Breakdown **1. SYN (Synchronization)** ``` Client → Server - Sets SYN flag = 1 - Chooses initial sequence number (ISN): seq=x - Typically sends MSS (Maximum Segment Size) option - No data payload ``` **2. SYN-ACK (Synchronization Acknowledgment)** ``` Server → Client - Sets SYN flag = 1, ACK flag = 1 - Chooses its own ISN: seq=y - Acknowledges client's sequence: ack=x+1 - Also sends MSS option ``` **3. ACK (Acknowledgment)** ``` Client → Server - Sets ACK flag = 1 - Sequence: seq=x+1 (next byte to send) - Acknowledgment: ack=y+1 (next byte expected) - Connection is now ESTABLISHED on both sides ``` ## Connection Termination (Four-Way Handshake) ``` Client Server | | |------------ FIN (seq=x) ------------->| | (FIN_WAIT_1 state) | | (CLOSE_WAIT) | | |<---------- ACK (ack=x+1) --------------| | (FIN_WAIT_2 state) | | | |<---------- FIN (seq=y) --------| | (TIME_WAIT state) (LAST_ACK) | | |---------- ACK (ack=y+1) ------->| | (CLOSED after timeout) (CLOSED) | | ``` ### Termination Steps **1. FIN from Client** ``` - FIN flag = 1 - Client stops sending data - Client can still receive data ``` **2. ACK from Server** ``` - Acknowledges the FIN - Server can still send remaining data ``` **3. FIN from Server** ``` - Server sends its own FIN when done - Server stops sending data ``` **4. ACK from Client** ``` - Acknowledges server's FIN - Enters TIME_WAIT state (default 2 minutes) - Ensures delayed packets are discarded ``` ## Key Protocol Details ### Sequence Numbers ```python # Sequence numbers track byte positions Initial Sequence Number (ISN) = random 32-bit value Each data byte increments the sequence number FIN and SYN flags consume one sequence number each Example: SYN: seq=1000 Data (10 bytes): seq=1001-1010 FIN: seq=1011 ``` ### Flags in TCP Header ``` |U|A|P|R|S|F| |R|C|S|S|Y|I| |G|K|H|T|N|N| ``` - **SYN**: Synchronize sequence numbers - **ACK**: Acknowledgment field is valid - **FIN**: No more data from sender - **RST**: Reset connection (error condition) - **PSH**: Push data to application immediately - **URG**: Urgent data present ### Sequence Number Validation ```python # Receiver validates incoming packets if received_seq == expected_seq: # In-order packet, process it expected_seq += len(data) else: # Out-of-order, may buffer or discard # Send ACK with expected sequence number ``` ## State Machine (Simplified) ``` Client Side: CLOSED → SYN_SENT → ESTABLISHED → FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT → CLOSED Server Side: LISTEN → SYN_RECEIVED → ESTABLISHED → CLOSE_WAIT → LAST_ACK → CLOSED ``` ## Practical Example with Wireshark-style Output ``` Frame 1: Client → Server [SYN] Seq=0, Ack=0, Len=0, Flags=SYN Frame 2: Server → Client [SYN, ACK] Seq=0, Ack=1, Len=0, Flags=SYN,ACK Frame 3: Client → Server [ACK] Seq=1, Ack=1, Len=0, Flags=ACK Frame 4: Client → Server [PSH, ACK] (HTTP GET) Seq=1, Ack=1, Len=100, Flags=PSH,ACK Frame 5: Server → Client [ACK] Seq=1, Ack=101, Len=0, Flags=ACK Frame 6: Server → Client [PSH, ACK] (HTTP Response) Seq=1, Ack=101, Len=500, Flags=PSH,ACK Frame 7: Client → Server [FIN, ACK] Seq=101, Ack=501, Len=0, Flags=FIN,ACK Frame 8: Server → Client [ACK] Seq=501, Ack=102, Len=0, Flags=ACK ``` ## Why TIME_WAIT? ``` 1. Ensures delayed packets from old connection don't interfere with new connection using same port 2. Allows final ACK to be retransmitted if lost 3. Default duration: 2 × MSL (Maximum Segment Lifetime) - MSL typically 30-120 seconds - TIME_WAIT = 60-240 seconds ``` This design ensures reliable, ordered delivery and graceful connection management.

Async scripts don't block --> <script async src="async.js"></script> </body> </html> ``` ### Execution Order ```javascript // Synchronous (default) <script>console.log(1)</script> <script>console.log(2)</script> // Output: 1, 2 (in order, blocks parsing) // Async <script async src="a.js"></script> <script async src="b.js"></script> // Output: unpredictable (loads in parallel, executes when ready) // Defer <script defer src="a.js"></script> <script defer src="b.js"></script> // Output: a, b (in order, after parsing complete) ``` ## Practical Examples ### SOP in Action ```javascript // Page: https://example.com // ✅ Works - same origin const data = await fetch('/api/users'); Survives forwarding better than SPF. **Limitation:** Doesn't specify *policy*—just proves authenticity. A legitimate key could theoretically be used maliciously. ## DMARC (Domain-based Message Authentication, Reporting & Conformance) **What it does:** Ties SPF and DKIM together and sets enforcement policy. **How:** You publish a policy saying "require SPF *and* DKIM" or "just one is fine." ACK (Acknowledgment)** ``` Client → Server - Sets ACK flag = 1 - Echoes server's sequence: ack=y+1 - Can include data payload (piggybacking) - Connection is now ESTABLISHED on both sides ``` ## Connection Termination (Four-Way Handshake) ``` Client Server | | |------------ FIN (seq=x) ------------->| | (FIN_WAIT_1 state) | | (CLOSE_WAIT) | | |<----- ACK (seq=y, ack=x+1) ------------| | (FIN_WAIT_2 state) | | | |<----- FIN (seq=y, ack=x+1) ------------| | (TIME_WAIT state) (LAST_ACK) | | |------------ ACK (seq=x+1, ack=y+1) -->| | (CLOSED state) (CLOSED) | | ``` ### FIN Packet Details **FIN (Finish)** ``` - Sets FIN flag = 1 - Indicates no more data will be sent - Still requires acknowledgment - Can include final data payload ``` **Key Points:** - FIN is a control signal, not a command to close immediately - Receiver can still send data after receiving FIN - Both sides must send FIN to fully close - TIME_WAIT prevents delayed packets from affecting new connections ## TCP Flags (Header Bits) ``` Bit Position: 0 1 2 3 4 5 Flag Name: FIN SYN RST PSH ACK URG Common Combinations: - SYN: 0 1 0 0 0 0 (0x02) - SYN-ACK: 0 1 0 0 1 0 (0x12) - ACK: 0 0 0 0 1 0 (0x10) - FIN-ACK: 1 0 0 0 1 0 (0x11) - RST: 0 0 1 0 0 0 (0x04) ``` ## Sequence & Acknowledgment Numbers ``` Purpose: Ensure reliable, ordered delivery Sequence Number (seq): - Identifies the byte position of the first data byte - Incremented by number of bytes sent - Initialized randomly for security (ISN) Acknowledgment Number (ack): - Confirms receipt of data up to (ack - 1) - Only valid when ACK flag is set - Indicates next expected sequence number ``` ## Practical Example with Wireshark-style Output ``` Frame 1: Client → Server [SYN] TCP: Flags=SYN, Seq=1000, Ack=0, Win=65535 Frame 2: Server → Client [SYN, ACK] TCP: Flags=SYN,ACK, Seq=2000, Ack=1001, Win=65535 Frame 3: Client → Server [ACK] TCP: Flags=ACK, Seq=1001, Ack=2001, Win=65535 [Data transfer happens here] Frame N: Client → Server [FIN, ACK] TCP: Flags=FIN,ACK, Seq=1500, Ack=2500 Frame N+1: Server → Client [ACK] TCP: Flags=ACK, Seq=2500, Ack=1501 Frame N+2: Server → Client [FIN, ACK] TCP: Flags=FIN,ACK, Seq=2500, Ack=1501 Frame N+3: Client → Server [ACK] TCP: Flags=ACK, Seq=1501, Ack=2501 ``` ## Why This Design? ✅ **Reliability**: Sequence numbers detect lost/duplicate packets ✅ **Ordering**: Packets reassembled in correct order ✅ **Flow Control**: Window size prevents overwhelming receiver ✅ **Security**: Random ISN prevents sequence number prediction attacks ✅ **Graceful Closure**: Both sides confirm connection termination This design makes TCP connection-oriented and reliable, unlike UDP's connectionless approach.

SYN (Client → Server)** ``` - Client sends SYN packet with: - SYN flag = 1 - Sequence number = x (random initial value) - No data payload - Client enters SYN_SENT state ``` **2. SYN-ACK (Server → Client)** ``` - Server responds with: - SYN flag = 1 (server also initiating its sequence) - ACK flag = 1 (acknowledging client's sequence) - Sequence number = y (server's random initial value) - Acknowledgment number = x + 1 (confirming receipt) - Server enters SYN_RECEIVED state ``` **3. ACK (Client → Server)** ``` - Client sends final ACK: - SYN flag = 0 - ACK flag = 1 - Sequence number = x + 1 - Acknowledgment number = y + 1 - Both sides now in ESTABLISHED state ``` ## Why Three Steps? - **Prevents spoofing**: Both sides prove they can receive packets - **Sequence synchronization**: Each side knows the other's starting sequence number - **Prevents old connections**: Random initial sequence numbers prevent confusion with old packets --- ## Connection Termination (Four-Way Handshake) Closing is more complex because either side can initiate: ``` Client Server | | |------------ FIN (seq=x) ------------->| | (FIN_WAIT_1 state) | | (CLOSE_WAIT) | | |<----- ACK (seq=y, ack=x+1) ------------| | (FIN_WAIT_2 state) | | | |<----- FIN (seq=y, ack=x+1) ------------| | (TIME_WAIT state) (LAST_ACK) | | |------- ACK (seq=x+1, ack=y+1) ------->| | (CLOSED state) (CLOSED) | | ``` ### Termination Steps **1. FIN (Initiator → Receiver)** ``` - FIN flag = 1 - Sequence number = current sequence - Indicates no more data to send - Initiator enters FIN_WAIT_1 ``` **2. ACK (Receiver → Initiator)** ``` - ACK flag = 1 - Acknowledges the FIN - Receiver enters CLOSE_WAIT - Initiator moves to FIN_WAIT_2 ``` **3. FIN (Receiver → Initiator)** ``` - Receiver sends its own FIN when ready - Receiver enters LAST_ACK state ``` **4.
